## Overview

The **Parallel Scheduling Pass** (`final-mvp-parallel-scheduling`) transforms sequential `scf.for` loops into parallel `scf.parallel` operations, enabling **multi-core execution**. The pass:

- **Analyzes data dependencies** to ensure safe parallelization
- **Validates disjoint memory writes** using IV-based subview analysis
- **Privatizes accumulators** for reduction patterns
- **Collapses nested parallel loops** into multi-dimensional parallel regions
- **Interchanges Linalg loops** for better cache locality
- **Hoists allocations** out of parallel regions

<Note>
This pass operates on **bufferized, vectorized IR** and should run near the end of the compilation pipeline, after tiling and vectorization.
</Note>

## What Gets Parallelized?

### SCF Loop Parallelization (Lines 568-627)

The pass targets **outer tile loops** from Linalg tiling:

```mlir
// Before: Sequential tile iteration
scf.for %i = %c0 to %c512 step %c128 {  // ← Parallelizable
  scf.for %ii = %c0 to %c128 step %c32 {  // ← Keep sequential (inner tile)
    linalg.matmul ins(...) outs(...)
  }
}

// After: Parallel tile iteration
scf.parallel (%i) = (%c0) to (%c512) step (%c128) {  // ✅ Parallel
  scf.for %ii = %c0 to %c128 step %c32 {  // Sequential (vectorized)
    linalg.matmul ins(...) outs(...)
  }
}
```

**Criteria** (lines 174-196):
1. No loop-carried dependencies (`iter_args`)
2. Constant bounds and step size
3. Simple yield (no reduction operands)
4. Trip count ≥ 2

### Linalg Generic Parallelization (Lines 632-778)

The pass converts high-level `linalg.generic` operations with parallel iterators:

```mlir
// Before: Linalg generic with parallel iterators
%result = linalg.generic {
  indexing_maps = [...],
  iterator_types = ["parallel", "parallel"]  // All parallel
} ins(%input) outs(%init) {
^bb0(%in: f32, %out: f32):
  %transformed = arith.mulf %in, %c2 : f32
  linalg.yield %transformed : f32
} -> tensor<128x128xf32>

// After: Lowered to parallel loops
scf.parallel (%i, %j) = (%c0, %c0) to (%c128, %c128) step (%c1, %c1) {
  %val = memref.load %input[%i, %j]
  %result = arith.mulf %val, %c2 : f32
  memref.store %result, %output[%i, %j]
}
```

## Safety Analysis

### SCF Safety Checks (Lines 208-272)

The pass validates loop body for parallel-safe operations:

```cpp
static bool isSafeToParallelizeSCF(scf::ForOp forOp) {
  if (!isParallelizableTileFor(forOp)) return false;

  bool ok = true;
  forOp.walk([&](Operation *op) {
    // Allow common non-side-effecting ops by dialect or type.
    if (isa<arith::ArithDialect>(op->getDialect()) ||
        isa<math::MathDialect>(op->getDialect()))
      return WalkResult::advance();
    
    if (isa<memref::LoadOp, memref::SubViewOp, memref::ViewOp,
            memref::ReinterpretCastOp, memref::CollapseShapeOp,
            memref::ExpandShapeOp, memref::AllocOp, memref::DeallocOp,
            vector::TransferReadOp, vector::TransferWriteOp,
            vector::ContractionOp>(op))
      return WalkResult::advance();

    // Reject calls and atomics
    if (isa<func::CallOp, func::CallIndirectOp>(op) ||
        isa<memref::AtomicRMWOp>(op)) {
      ok = false;
      return WalkResult::interrupt();
    }

    // Check memory effects
    if (auto iface = dyn_cast<MemoryEffectOpInterface>(op)) {
      SmallVector<MemoryEffects::EffectInstance> effects;
      iface.getEffects(effects);
      bool hasWrite = llvm::any_of(effects, [](const auto &eff) {
        return isa<MemoryEffects::Write>(eff.getEffect());
      });
      if (!hasWrite)
        return WalkResult::advance();
      if (isa<memref::StoreOp>(op))
        return WalkResult::advance();
      
      ok = false;
      return WalkResult::interrupt();
    }
    // Unknown op: reject
    ok = false;
    return WalkResult::interrupt();
  });
  return ok;
}
```

**Allowed Operations**:
- ✅ Arithmetic (`arith.addf`, `arith.mulf`, etc.)
- ✅ Math functions (`math.sqrt`, `math.exp`, etc.)
- ✅ Memory reads (`memref.load`, `vector.transfer_read`)
- ✅ Memory writes to **disjoint regions** (`memref.store`, `vector.transfer_write`)
- ✅ View operations (`memref.subview`, `memref.cast`)

**Rejected Operations**:
- ❌ Function calls (unknown side effects)
- ❌ Atomic operations (implies shared state)
- ❌ Operations with unanalyzable memory effects

### Disjoint Write Analysis (Lines 274-323)

The pass verifies that writes in different loop iterations access non-overlapping memory:

```cpp
static bool isDisjointSubviewForIV(Value memref, Value iv, int64_t stepVal) {
  // Trace through view-like ops; count exactly one IV in subview offsets and
  // require the tile size for that dim to match the loop step.
  Value cur = memref;
  int ivCount = 0;
  while (cur) {
    if (auto sub = cur.getDefiningOp<memref::SubViewOp>()) {
      auto offsets = sub.getMixedOffsets();
      auto sizes = sub.getMixedSizes();
      for (auto it : llvm::enumerate(offsets)) {
        if (auto v = it.value().dyn_cast<Value>()) {
          if (v == iv) {
            ivCount++;
            auto sz = sizes[it.index()];
            if (auto szAttr = sz.dyn_cast<Attribute>()) {
              if (auto intAttr = szAttr.dyn_cast<IntegerAttr>()) {
                if (intAttr.getInt() != stepVal) return false;
              }
            }
          }
        }
      }
      cur = sub.getSource();
      continue;
    }
    // ... handle other view ops ...
    break;
  }
  return ivCount == 1;  // IV must appear exactly once in offset
}
```

**Example of Disjoint Writes**:
```mlir
// Loop with step=32
scf.for %i = %c0 to %c512 step %c32 {
  // Extract 32-element slice starting at %i
  %tile = memref.subview %buffer[%i][32][1] : memref<512xf32>
  
  // Write to tile (writes to %buffer[%i:%i+32])
  vector.transfer_write %vec, %tile[%c0] : vector<32xf32>, memref<32xf32>
}

// ✅ Disjoint: Iteration 0 writes [0:32], Iteration 1 writes [32:64], etc.
```

**Example of Overlapping Writes (Rejected)**:
```mlir
scf.for %i = %c0 to %c512 step %c32 {
  // Slice of size 64 (larger than step)
  %tile = memref.subview %buffer[%i][64][1] : memref<512xf32>
  
  // Write to tile (writes to %buffer[%i:%i+64])
  vector.transfer_write %vec, %tile[%c0] : vector<64xf32>, memref<64xf32>
}

// ❌ Overlapping: Iteration 0 writes [0:64], Iteration 1 writes [32:96]
//                  Overlap at [32:64] - NOT SAFE TO PARALLELIZE
```

## Privatization (Lines 328-445)

For accumulation patterns, the pass **privatizes** reduction buffers:

### Example: Parallel Reduction

<Tabs>
  <Tab title="Before Privatization">
    ```mlir
    func.func @parallel_reduce(%input: memref<512x128xf32>,
                               %output: memref<512x128xf32>) {
      %c0 = arith.constant 0 : index
      %c32 = arith.constant 32 : index
      %c512 = arith.constant 512 : index
      
      // Sequential accumulation
      scf.for %i = %c0 to %c512 step %c32 {
        %tile_input = memref.subview %input[%i, 0][32, 128][1, 1]
          : memref<512x128xf32> to memref<32x128xf32>
        
        // Read current accumulator
        %vec_acc = vector.transfer_read %output[%i, 0], %c0_f32
          : memref<512x128xf32>, vector<32x128xf32>
        
        %vec_in = vector.transfer_read %tile_input[0, 0], %c0_f32
          : memref<32x128xf32>, vector<32x128xf32>
        
        // Accumulate
        %sum = arith.addf %vec_acc, %vec_in : vector<32x128xf32>
        
        // Write back (creates dependency on previous iteration)
        vector.transfer_write %sum, %output[%i, 0]
          : vector<32x128xf32>, memref<512x128xf32>
      }
    }
    ```
    
    **Issue**: Each iteration reads and updates `%output`, creating a **loop-carried dependency**.
  </Tab>
  <Tab title="After Privatization">
    ```mlir
    func.func @parallel_reduce(%input: memref<512x128xf32>,
                               %output: memref<512x128xf32>) {
      %c0 = arith.constant 0 : index
      %c32 = arith.constant 32 : index
      %c512 = arith.constant 512 : index
      %c0_f32 = arith.constant 0.0 : f32
      
      // Parallel accumulation with privatization
      scf.parallel (%i) = (%c0) to (%c512) step (%c32) {
        
        // Private accumulator (per-thread, zero-initialized)
        %priv_acc = memref.alloc() : memref<32x128xf32>
        linalg.fill ins(%c0_f32) outs(%priv_acc) : memref<32x128xf32>
        
        %tile_input = memref.subview %input[%i, 0][32, 128][1, 1]
          : memref<512x128xf32> to memref<32x128xf32>
        
        // Accumulate into private buffer
        %vec_in = vector.transfer_read %tile_input[0, 0], %c0_f32
          : memref<32x128xf32>, vector<32x128xf32>
        vector.transfer_write %vec_in, %priv_acc[0, 0]
          : vector<32x128xf32>, memref<32x128xf32>
        
        // Critical section: Reduce private buffer into shared output
        %priv_vec = vector.transfer_read %priv_acc[0, 0], %c0_f32
          : memref<32x128xf32>, vector<32x128xf32>
        %out_vec = vector.transfer_read %output[%i, 0], %c0_f32
          : memref<512x128xf32>, vector<32x128xf32>
        %sum = arith.addf %priv_vec, %out_vec : vector<32x128xf32>
        vector.transfer_write %sum, %output[%i, 0]
          : vector<32x128xf32>, memref<512x128xf32>
      }
    }
    ```
    
    **Benefits**:
    - ✅ **No inter-iteration dependencies** during main computation
    - ✅ **Parallel accumulation** into private buffers
    - ✅ **Single atomic reduction** at the end (often auto-parallelized by backend)
  </Tab>
</Tabs>

## Loop Collapsing (Lines 447-488)

The pass collapses nested `scf.parallel` + `scf.for` patterns:

```mlir
// Before: Parallel outer, sequential inner
scf.parallel (%i) = (%c0) to (%c512) step (%c64) {  // 8 parallel tasks
  scf.for %j = %c0 to %c512 step %c64 {  // 8 sequential iterations per task
    // 64x64 tile computation
  }
}
// Total parallelism: 8 tasks

// After: Collapsed to 2D parallel
scf.parallel (%i, %j) = (%c0, %c0) to (%c512, %c512) step (%c64, %c64) {
  // 64x64 tile computation
}
// Total parallelism: 64 tasks (8×8)
```

**Profitability**: Collapsing increases parallelism from `N` to `N×M`, improving scalability on many-core systems.

## Linalg Loop Interchange (Lines 642-680)

The pass reorders loops to place the **contiguous dimension innermost**:

```mlir
// Before: Suboptimal loop order
linalg.generic {
  indexing_maps = [affine_map<(d0, d1) -> (d0, d1)>,  // Input
                   affine_map<(d0, d1) -> (d0, d1)>], // Output
  iterator_types = ["parallel", "parallel"]
} ins(%in : memref<512x512xf32>) outs(%out : memref<512x512xf32>) {
  // Loop order: d0 (outer), d1 (inner)
  // Memory access: row-major, but d1 is fastest-changing
}

// Heuristic: Last dimension of output map should be innermost
affine_map<(d0, d1) -> (d0, d1)>  // d1 is last
// Current loop order: d0, d1  ✅ Already optimal

// After interchange (if needed):
linalg.generic {
  indexing_maps = [affine_map<(d1, d0) -> (d0, d1)>,  // Interchanged
                   affine_map<(d1, d0) -> (d0, d1)>],
  iterator_types = ["parallel", "parallel"]
} // Loop order: d1 (outer), d0 (inner) for better cache locality
```

## Performance Impact

### Scalability Analysis

| Matrix Size | Tiles (128×128) | Parallel Tasks | Sequential Work | Ideal Speedup (16 cores) | Actual Speedup |
|-------------|-------------------|----------------|-----------------|--------------------------|----------------|
| 512×512 | 4×4×4 = 64 | 16 (4×4 M×N) | 4 (K loop) | 16× | 12-14× (75-88%) |
| 1024×1024 | 8×8×8 = 512 | 64 (8×8 M×N) | 8 (K loop) | 16× | 14-15× (88-94%) |
| 2048×2048 | 16×16×16 = 4096 | 256 (16×16 M×N) | 16 (K loop) | 16× | 15-16× (94-100%) |

### Why Not 100% Scaling?

<Steps>
  <Step title="Amdahl's Law">
    The **K loop remains sequential** (reduction into C), limiting parallel fraction:
    
    - Parallel: M×N tile computation (75-90% of time)
    - Sequential: K-dimension reduction (10-25% of time)
    - **Theoretical max speedup**: ~10-16× on 16 cores
  </Step>
  <Step title="Synchronization Overhead">
    - `scf.parallel` insert implicit barriers
    - Reduction into shared `%C` requires atomic/lock operations
    - Overhead: 2-5% of execution time
  </Step>
  <Step title="Memory Bandwidth">
    - 16 cores saturate memory bandwidth (~100 GB/s)
    - Cache coherency traffic increases with core count
    - NUMA effects on multi-socket systems
  </Step>
  <Step title="Load Imbalance">
    - Partial tiles at matrix edges complete faster
    - Some threads idle while others finish last tiles
    - Mitigation: Dynamic scheduling (not implemented)
  </Step>
</Steps>

## Troubleshooting

### Issue: Loop not parallelized despite being safe

**Symptom**:
```
[DEBUG] Skip scf.for: no nested loop or no vector activity
```

**Cause**: Pass requires both inner loops AND vector operations (lines 606-609).

**Solution**: Ensure vectorization ran before parallel scheduling:
```bash
stablehlo-opt \
  --toy-vectorize \  # Must run first
  --final-mvp-parallel-scheduling \
  input.mlir
```

### Issue: "Non-constant bounds/step" warning

**Cause**: Loop bounds are dynamic (lines 179-185).

**Solution**: Tile with static sizes before parallelization:
```bash
--stablehlo-linalg-tiling="l2-tile-sizes=1,128,128"  # Static 128×128 tiles
```

### Issue: Race conditions / wrong results

**Symptom**: Output differs between sequential and parallel runs.

**Debugging**:
```bash
# Run with ThreadSanitizer
clang -fsanitize=thread -g parallel_code.ll -o program
./program
```

**Common Causes**:
1. **False negative in disjoint analysis**: Subview slices actually overlap
2. **Untracked side effects**: Custom ops with hidden shared state
3. **Incorrect privatization**: Accumulator not properly per-thread

### Issue: Performance degradation with parallelization

**Symptom**: Parallel code slower than sequential.

**Possible Causes**:

1. **Too many small tasks**:
   ```mlir
   // Bad: 4096 tasks of 1×1 each
   scf.parallel (%i, %j) = (%c0, %c0) to (%c64, %c64) step (%c1, %c1)
   
   // Good: 16 tasks of 16×16 each
   scf.parallel (%i, %j) = (%c0, %c0) to (%c64, %c64) step (%c16, %c16)
   ```
   
   **Fix**: Increase tile sizes.

2. **False sharing**:
   ```mlir
   // Adjacent tiles write to same cache line (64 bytes)
   scf.parallel (%i) = (%c0) to (%c512) step (%c4) {  // 4-element tiles
     memref.store %val, %buf[%i] : memref<512xf32>  // 4 bytes per element
   }
   // Fix: Tile by cache line size (16 f32 elements)
   ```

3. **Memory bandwidth saturation**:
   ```bash
   # Check with perf
   perf stat -e cycles,instructions,cache-misses,LLC-load-misses ./program
   ```
