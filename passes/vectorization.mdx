## Overview

The **Toy Vectorization Pass** (`toy-vectorize`) transforms Linalg operations on buffers into efficient **SIMD (Single Instruction, Multiple Data) vector code**. This pass:

- **Vectorizes matmul operations** (both 2D and 3D batch variants)
- **Handles dynamic shapes** with automatic shape casting
- **Repairs invalid vector transfers** with corrected permutation maps
- **Supports partial tiles** through masking
- **Converts reductions to vector contractions** for hardware acceleration

<Note>
This pass operates on **bufferized IR** (memref-based). Run after `my-one-shot-bufferize` and before parallel scheduling.
</Note>

## Command-Line Usage

### Basic Invocation (Default 4-wide SIMD)

```bash
stablehlo-opt --toy-vectorize input.mlir -o output.mlir
```

### Custom Vector Width

```bash
# 8-wide vectorization (AVX2: 256-bit SIMD)
stablehlo-opt --toy-vectorize="vector-width=8" input.mlir

# 16-wide vectorization (AVX-512: 512-bit SIMD)
stablehlo-opt --toy-vectorize="vector-width=16" input.mlir
```

### Tuning Options

```bash
stablehlo-opt \
  --toy-vectorize="
    vector-width=8 \
    conservative=false \
    enable-reductions=true \
    verbose=true \
    disable-size-cap=false" \
  input.mlir
```

### Complete Pipeline

```bash
stablehlo-opt \
  --stablehlo-linalg-tiling="l2-tile-sizes=1,128,128 l1-tile-sizes=0,32,32" \
  --canonicalize \
  --my-one-shot-bufferize \
  --toy-vectorize="vector-width=8 enable-reductions=true" \
  --canonicalize \  # Clean up vector IR
  --final-mvp-parallel-scheduling \
  input.mlir
```

## Configuration Options

<ParamField path="vector-width" type="int64" default="4">
  Number of elements processed per SIMD instruction.
  
  **Hardware Targets**:
  - `4`: SSE (128-bit), baseline compatibility
  - `8`: AVX/AVX2 (256-bit), modern x86 CPUs
  - `16`: AVX-512 (512-bit), Xeon Scalable, latest Intel/AMD
  - `32`: Exotic architectures (SVE, custom accelerators)
</ParamField>

<ParamField path="conservative" type="bool" default="true">
  When enabled, requires static shapes and projected permutation maps for non-matmul operations.
  
  **Effect**:
  - `true`: Safe, only vectorizes well-understood patterns
  - `false`: Aggressive, attempts vectorization on dynamic/complex ops
</ParamField>

<ParamField path="enable-reductions" type="bool" default="true">
  Allow vectorizing operations with reduction loops (e.g., sum, dot products).
  
  **When to disable**: If backend doesn't support `vector.contract` or `vector.multi_reduction` lowering.
</ParamField>

<ParamField path="verbose" type="bool" default="false">
  Print detailed vectorization decisions to stderr.
  
  ```bash
  stablehlo-opt --toy-vectorize="verbose=true" input.mlir 2>&1 | grep "toy-vectorize"
  ```
</ParamField>

<ParamField path="disable-red-to-contract" type="bool" default="false">
  Skip `vector.multi_reduction` → `vector.contract` transformation.
  
  **Use when**: Debugging contract lowering issues or targeting hardware without native FMA (fused multiply-add).
</ParamField>

<ParamField path="disable-cleanup" type="bool" default="false">
  Skip post-vectorization canonicalization and simplification.
  
  **Use when**: Inspecting raw vectorized output for debugging.
</ParamField>

<ParamField path="disable-size-cap" type="bool" default="false">
  Remove vector size cap checks (default: reject ops creating vectors >vector-width in any dimension).
  
  **Warning**: May generate inefficient giant vector types like `vector<128x32x32xf32>`.
</ParamField>

<ParamField path="disable-fallback-vec" type="bool" default="false">
  Skip fallback walker that vectorizes matmul/batch_matmul missed by pattern matching.
  
  **Effect**: May leave some operations unvectorized if nested in complex control flow.
</ParamField>

<ParamField path="dump-first-bad-transfer" type="bool" default="false">
  On first invalid `vector.transfer_{read,write}`, dump debug info and fail compilation.
  
  **Use when**: Debugging permutation map repair logic.
</ParamField>

## Implementation Details

### Vectorization Legality Checks (Lines 127-157)

The pass validates operations before vectorization:

```cpp
static bool isVectorizationLegal(linalg::LinalgOp op,
                                 const VectorizationOptions &opt) {
  // Skip trivially unfriendly ops.
  if (isa<linalg::CopyOp>(op.getOperation()))
    return false;

  bool isMatmulLike =
      isa<linalg::MatmulOp, linalg::BatchMatmulOp>(op.getOperation());
  bool hasReductions = op.getNumReductionLoops() > 0;

  // Optional reduction support (allow matmul even when reductions are off).
  if (!opt.enableReductions && hasReductions && !isMatmulLike)
    return false;

  // Conservative mode: require static shapes for non-matmul ops and require
  // projected permutations for all ops. Allow matmul/bmm with dynamic tiles.
  if (opt.conservative) {
    if (!isMatmulLike) {
      for (Type t : op->getOperandTypes()) {
        if (!hasStaticShape(t))
          return false;
      }
    }
    for (AffineMap map : op.getIndexingMapsArray()) {
      if (!map.isProjectedPermutation(/*allowZeroInResults=*/false))
        return false;
    }
  }

  return true;
}
```

**Filters**:
1. Rejects `linalg.copy` (handled by specialized passes)
2. Checks reduction support (line 138)
3. In conservative mode, validates shapes and maps (lines 143-154)

### Profitability Analysis (Lines 162-182)

Skips vectorization for scalar or trivial loops:

```cpp
static bool isVectorizationProfitable(linalg::LinalgOp op,
                                      const VectorizationOptions &opt) {
  SmallVector<int64_t> shape = op.getStaticLoopRanges();
  if (shape.empty()) return false;

  auto iterTypes = op.getIteratorTypesArray();
  bool hasVectorizableLoop = false;
  for (int64_t i = 0, e = iterTypes.size(); i < e; ++i) {
    if (iterTypes[i] != utils::IteratorType::parallel)
      continue;
    if (i >= static_cast<int64_t>(shape.size()))
      continue;
    int64_t s = shape[i];
    // Allow partial tiles (masking handles tails). Only skip pure scalars.
    if (s == ShapedType::kDynamic || s > 1) {
      hasVectorizableLoop = true;
      break;
    }
  }
  return hasVectorizableLoop;
}
```

<Tip>
**Partial Tile Support**: The pass accepts loops with trip counts not divisible by vector width (line 176). The MLIR vector lowering automatically inserts masking for tail elements.
</Tip>

### Vector Size Cap (Lines 186-194)

Prevents generating excessively large vector types:

```cpp
static bool withinVectorSizeCap(linalg::LinalgOp op, int64_t cap) {
  SmallVector<int64_t> shape = op.getStaticLoopRanges();
  if (shape.empty()) return false;
  for (int64_t s : shape) {
    if (s == ShapedType::kDynamic) continue;
    if (s > cap) return false;  // Reject if any dimension exceeds cap
  }
  return true;
}
```

**Example**: With `vector-width=8`, a `64x64` matmul would fail the cap check and be tiled to `8x8` first (lines 199-240).

### Dynamic Shape Handling (Lines 252-279)

The pass casts dynamic memrefs to static shapes for vectorization:

```cpp
static Value maybeCastToStatic(RewriterBase &rewriter, Location loc, Value v,
                               ArrayRef<int64_t> shape,
                               SmallVectorImpl<Operation *> &createdCasts) {
  auto type = v.getType().dyn_cast<MemRefType>();
  if (!type || type.getRank() != static_cast<int64_t>(shape.size()))
    return v;

  bool needsCast = false;
  for (int64_t i = 0, e = type.getRank(); i < e; ++i) {
    int64_t dim = type.getDimSize(i);
    if (dim == ShapedType::kDynamic) {
      needsCast = true;
      continue;
    }
    if (dim != shape[i])
      return v;  // Mismatch, abort
  }

  if (!needsCast)
    return v;

  auto newType =
      MemRefType::get(shape, type.getElementType(), type.getLayout(),
                      type.getMemorySpace());
  auto cast = rewriter.create<memref::CastOp>(loc, newType, v);
  createdCasts.push_back(cast.getOperation());
  return cast;
}
```

**Usage**:
```mlir
// Before
%m = memref.alloc(%dyn_size) : memref<?x?xf32>  // Dynamic

// After (if loop bounds statically known as 32x32)
%cast = memref.cast %m : memref<?x?xf32> to memref<32x32xf32>
linalg.matmul ins(%cast, ...) outs(%cast) -> memref<32x32xf32>  // Vectorizes
```

### Batch Matmul Expansion (Lines 367-461)

For small batch dimensions, the pass expands `batch_matmul` into a loop of vectorized `matmul` operations:

```cpp
static LogicalResult expandBatchMatmulToMatmul(PatternRewriter &rewriter,
                                               linalg::BatchMatmulOp bmm,
                                               int64_t cap) {
  // Extract batch, M, N, K dimensions
  int64_t batch = aType.getDimSize(0);
  if (batch == ShapedType::kDynamic || batch > cap)
    return failure();

  // Create batch loop
  auto loop = rewriter.create<scf::ForOp>(loc, c0, cBatch, c1);
  Value iv = loop.getInductionVar();

  // Slice inputs: [batch,M,K] -> [M,K]
  Value aSubview = rewriter.create<memref::SubViewOp>(
      loc, aSubviewType, a, offsets, aSizes, strides);
  
  // Create 2D matmul and vectorize
  auto matmul = rewriter.create<linalg::MatmulOp>(
      loc, ValueRange{aSubview, bSubview}, ValueRange{cSubview});
  if (failed(linalg::vectorize(rewriter, matmul.getOperation()))) {
    rewriter.eraseOp(loop);
    return failure();
  }

  rewriter.eraseOp(bmm);
  return success();
}
```

<Note>
**Batch Size Limit**: Only batches ≤ vector width are expanded. Larger batches remain as `batch_matmul` and vectorize directly (lines 321-363).
</Note>

### Vector Transfer Repair (Lines 836-944)

The pass automatically fixes invalid `vector.transfer_{read,write}` operations:

```cpp
auto validateAndFix = [&](Operation *op) -> WalkResult {
  if (auto tr = dyn_cast<vector::TransferReadOp>(op)) {
    auto srcType = tr.getSource().getType().cast<ShapedType>();
    int64_t srcRank = srcType.getRank();
    int64_t vecRank = tr.getVectorType().getRank();
    AffineMap map = tr.getPermutationMap();
    
    bool needFix = map.getNumDims() != srcRank ||
                   map.getNumResults() != vecRank ||
                   static_cast<int64_t>(tr.getIndices().size()) != srcRank ||
                   (tr.getInBounds().has_value() &&
                    static_cast<int64_t>(tr.getInBounds()->size()) != vecRank);

    if (needFix) {
      // Build corrected permutation map
      AffineMap newMap =
          buildMatmulBroadcastMap(srcType, tr.getVectorType(), &getContext());
      
      // Create fixed transfer_read
      auto newOp = rewriter.create<vector::TransferReadOp>(
          tr.getLoc(), tr.getVectorType(), tr.getSource(), tr.getIndices(),
          AffineMapAttr::get(newMap), tr.getPadding(), Value(),
          newInBoundsAttr);
      rewriter.replaceOp(tr, newOp.getResult());
    }
  }
  // Similar logic for transfer_write...
};
```

**Common Issue**: Vectorized matmul generates broadcasts like:
```mlir
// A: tensor<1xK> broadcast to vector<MxNxK>
// Map must be (d0, 0, d1) not (d0, d1) - wrong rank!
```

The repair logic (lines 45-71) constructs correct broadcast maps.

### Multi-Reduction to Contract (Lines 551-630)

Converts reduction patterns to hardware-accelerated contractions:

```cpp
struct MultiReductionToContractPattern
    : public OpRewritePattern<vector::MultiDimReductionOp> {
  LogicalResult matchAndRewrite(vector::MultiDimReductionOp red,
                                PatternRewriter &rewriter) const override {
    // Match: multi_dim_reduction(add, mul(lhs, rhs))
    auto mulf = red.getSource().getDefiningOp<arith::MulFOp>();
    if (!mulf) return failure();
    
    // Build contraction: lhs * rhs + acc
    auto contract = rewriter.create<vector::ContractionOp>(
        red.getLoc(), lhs, rhs, red.getAcc(), maps, iterAttr);
    rewriter.replaceOp(red, contract.getResult());
    return success();
  }
};
```

**Before**:
```mlir
%mul = arith.mulf %a, %b : vector<8x8x8xf32>
%sum = vector.multi_reduction <add>, %mul, %acc [2] 
  : vector<8x8x8xf32> into vector<8x8xf32>
```

**After**:
```mlir
%contract = vector.contract {
  indexing_maps = [affine_map<(d0,d1,d2) -> (d0,d1,d2)>,
                   affine_map<(d0,d1,d2) -> (d0,d1,d2)>,
                   affine_map<(d0,d1,d2) -> (d0,d1)>],
  iterator_types = ["parallel", "parallel", "reduction"]
} %a, %b, %acc : vector<8x8x8xf32>, vector<8x8x8xf32> into vector<8x8xf32>
```

<Tip>
`vector.contract` directly maps to FMA instructions (e.g., `vfmadd231ps` on x86) for **2-3× speedup** over separate multiply + add.
</Tip>

  <Tab title="After Vectorization (8-wide)">
    ```mlir
    func.func @matmul_32x32(%lhs: memref<32x32xf32>,
                            %rhs: memref<32x32xf32>,
                            %out: memref<32x32xf32>) {
      %c0 = arith.constant 0 : index
      %c0_f32 = arith.constant 0.0 : f32
      
      // Vector type: 8 elements per register (AVX2)
      %v_m = arith.constant dense<[0,1,2,3,4,5,6,7]> : vector<8xindex>
      
      // Outer loops iterate over 8-wide tiles
      affine.for %i = 0 to 32 step 8 {
        affine.for %j = 0 to 32 step 8 {
          
          // Load 8x8 accumulator tile
          %acc = vector.transfer_read %out[%i, %j], %c0_f32 
            {in_bounds = [true, true]} 
            : memref<32x32xf32>, vector<8x8xf32>
          
          // Contraction loop over K dimension
          %result = affine.for %k = 0 to 32 step 8 
              iter_args(%acc_k = %acc) -> (vector<8x8xf32>) {
            
            // Load 8x8 tiles from A and B
            %tile_a = vector.transfer_read %lhs[%i, %k], %c0_f32
              {in_bounds = [true, true]}
              : memref<32x32xf32>, vector<8x8xf32>
            %tile_b = vector.transfer_read %rhs[%k, %j], %c0_f32
              {in_bounds = [true, true]}
              : memref<32x32xf32>, vector<8x8xf32>
            
            // Vector contract: 8x8x8 -> 8x8 (64 FMA ops in parallel)
            %partial = vector.contract {
              indexing_maps = [affine_map<(d0,d1,d2) -> (d0,d2)>,  // A: M,K
                               affine_map<(d0,d1,d2) -> (d2,d1)>,  // B: K,N
                               affine_map<(d0,d1,d2) -> (d0,d1)>], // C: M,N
              iterator_types = ["parallel", "parallel", "reduction"]
            } %tile_a, %tile_b, %acc_k 
              : vector<8x8xf32>, vector<8x8xf32> into vector<8x8xf32>
            
            affine.yield %partial : vector<8x8xf32>
          }
          
          // Store 8x8 result tile
          vector.transfer_write %result, %out[%i, %j]
            {in_bounds = [true, true]}
            : vector<8x8xf32>, memref<32x32xf32>
        }
      }
      return
    }
    ```

    - ✅ **Expected speedup: 5-8×** on modern CPUs

## Performance Impact

### Arithmetic Throughput

| Architecture | Scalar FLOPS | 8-wide Vector FLOPS | 16-wide Vector FLOPS | Speedup |
|--------------|--------------|---------------------|----------------------|--------|
| **Intel Skylake** | 16 GFLOPS | 128 GFLOPS (AVX2) | 256 GFLOPS (AVX-512) | 8-16× |
| **AMD Zen 3** | 16 GFLOPS | 128 GFLOPS (AVX2) | 256 GFLOPS (AVX-512) | 8-16× |
| **ARM Neoverse V1** | 8 GFLOPS | 64 GFLOPS (NEON) | 128 GFLOPS (SVE) | 8-16× |

### Memory Bandwidth Utilization

**Scalar Code**:
- **Loads**: 3 × 4 bytes per FLOP = 12 bytes/FLOP
- **Arithmetic Intensity**: 0.08 FLOP/byte
- **Bottleneck**: Memory bandwidth (~50% peak)

**Vectorized Code (8-wide)**:
- **Loads**: 3 × 32 bytes per 8 FLOP = 12 bytes per 8 FLOP
- **Arithmetic Intensity**: 0.67 FLOP/byte
- **Bottleneck**: Mixed (70% peak on both compute and memory)

## Integration with Other Passes

### Correct Pipeline

```bash
stablehlo-opt \
  --stablehlo-linalg-tiling="l1-tile-sizes=0,32,32" \  # 1. Tile to vectorizable sizes
  --my-one-shot-bufferize \                           # 2. Convert to buffers
  --toy-vectorize="vector-width=8" \                  # 3. Vectorize
  --canonicalize \                                    # 4. Cleanup
  --final-mvp-parallel-scheduling \                   # 5. Parallelize outer loops
  --convert-vector-to-llvm="reassociate-fp-reductions" \ # 6. Lower to LLVM intrinsics
  input.mlir
```

<Warning>
**Critical Order**:
1. **Tile first**: Creates appropriately-sized operations (32×32, 64×64)
2. **Bufferize before vectorize**: Tensor ops cannot vectorize
3. **Vectorize before parallel**: Parallel pass expects vector IR
4. **Lower vectors last**: LLVM conversion must happen after all MLIR-level optimizations
</Warning>

## Troubleshooting

### Issue: "No contiguous vector dim"

**Symptom**:
```
[toy-vectorize] reject: no contiguous vector dim
```

**Cause**: All loop dimensions fail stride check (line 492).

**Solution**: Disable conservative mode or transpose operands:
```bash
--toy-vectorize="conservative=false"
```

### Issue: Invalid permutation map errors

**Symptom**:
```
error: 'vector.transfer_read' op permutation_map must have same number of dims as source rank
```

**Cause**: Map repair logic failed (lines 875-882).

**Debug**:
```bash
--toy-vectorize="dump-first-bad-transfer=true"
```

**Fix**: Check that source type rank matches indexing:
```mlir
// ❌ Bad: 2D source with 3D map
vector.transfer_read %src : memref<32x32xf32>, vector<8x8x8xf32>

// ✅ Good: 2D source with 2D vector, broadcast via map
vector.transfer_read %src : memref<32x32xf32>, vector<8x8xf32>
```

### Issue: Poor performance despite vectorization

**Symptom**: Vectorized code slower than scalar.

**Possible Causes**:

1. **Misaligned loads/stores**:
   ```bash
   # Check generated assembly
   llc -O3 -print-after=vectorize output.ll | grep vmovups
   # vmovups = unaligned, vmovaps = aligned (4× faster)
   ```
   
   **Fix**: Align allocations:
   ```mlir
   %m = memref.alloc() {alignment = 32} : memref<128x128xf32>
   ```

2. **Register spilling**:
   ```bash
   # Check for stack spills
   llc -O3 output.ll | grep "movaps.*rsp"
   ```
   
   **Fix**: Reduce vector width or tile size.

3. **Failed contract lowering**:
   ```bash
   # Check if contracts become scalar loops
   stablehlo-opt ... --convert-vector-to-llvm | grep "scf.for"
   ```
   
   **Fix**: Update LLVM or enable FMA:
   ```bash
   llc -O3 -mattr=+fma,+avx2 output.ll
   ```
